Context-Aware Bug Localisation

Using CodeBERT for Pre-Commit Java Project Analysis

Demo Day

Introduction

This project explores whether a context-aware, classification-based CodeBERT model can effectively identify buggy lines in Java code before they are committed to production.. The core idea is to assist developers in detecting issues at the source level, during the pre-commit phase. It does this by leveraging the surrounding code context, commit messages, and prior code snippets (line changed).

The research is guided by the following question:
Can a context-aware, classification-based CodeBERT model effectively localise pre-commit Java bugs using source-level inputs, and how does its performance compare to recent bug localisation approaches?

The primary goals of the project are:

  • To fine-tune CodeBERT for line-level bug detection
  • To evaluate the model‚Äôs performance on dedicated test datasets
  • To compare metrics and evalute against prior bug localisation studies

I conducted a thorough review of recent academic studies and industry research to ground this project in the current state of bug localisation. The review focused on the evolution of bug localisation‚Äîfrom traditional static analysis tools to state-of-the-art Large Language Models (LLMs)‚Äîand informed my decision to use CodeBERT for pre-commit Java bug detection.

Large Language Models (LLMs)

LLMs like BERT, GPT, and CodeBERT are based on the Transformer architecture introduced by Vaswani et al. These models allow for parallel processing of code and natural language, capturing long-range dependencies and semantic meaning‚Äîcrucial for tasks like bug localisation. The ability to interpret both code syntax and developer-written commit messages makes them especially powerful for context-aware systems.

CodeBERT

CodeBERT, developed by Microsoft, is pre-trained on millions of code-text pairs and supports six programming languages, including Java. Its training objectives‚ÄîMasked Language Modelling (MLM) and Replaced Token Detection (RTD)‚Äîenable it to learn code semantics and spot subtle errors in code. Given its strong results on tasks like code summarisation, retrieval, and bug repair, CodeBERT was selected as the backbone of my classification model.

Previous Studies on Bug Localisation
  • Ciborowska & Damevski (2022) used BERT with commit-level encodings for bug localisation. Their work showed the importance of incorporating commit messages and diffs, which I extended further in my model‚Äôs context fields.
  • Mashhadi & Hemmati (2021) fine-tuned CodeBERT on ManySStuBs4J for bug repair and localisation. Their results showed high performance on repeated bugs but highlighted challenges on unique ones‚Äîdirectly influencing my focus on generalisation.
  • Zhang et al. (2024) proposed WELL, a weakly supervised localisation system. Their method demonstrates how useful coarse-grained binary labels can be when fine-grained annotations are missing‚Äîoffering future direction for scaling bug localisation systems.
  • RL-Locator (Chakraborty et al., 2024) introduced reinforcement learning to iteratively improve localisation. Though out of scope for this project, it showed the value of feedback-driven localisation systems for evolving codebases.
Summary

This literature review confirmed that while traditional static tools like SonarQube offer basic rule-based bug detection, they lack contextual awareness and are prone to false positives. Modern LLM-based methods outperform them by incorporating syntax, semantics, and context. CodeBERT‚Äôs dual-modality and extensibility made it the ideal model for my project‚Äôs goal: pre-commit, context-aware bug localisation for Java code.

Methodology

The ManySStuBs4J dataset was used, with its repetition and unique subsets providing two evaluation perspectives. The model was trained and validated solely on the repetition subset, which includes recurring bug patterns. The unique subset‚Äîwhich contained non-redundant, novel bugs‚Äîwas held out entirely and used only for independent testing to evaluate generalisation. All examples were enhanced with rich context, including commit messages, file history, and surrounding code.

üßæ Dataset Preparation

The ManySStuBs4J dataset came in two forms: repetition and unique subsets. These splits help evaluate both common bug patterns and generalisation to novel ones. All examples were enhanced with additional context, such as commit messages and nearby lines of code, making them suitable for input into a transformer-based classifier.

üì¶ Input Formatting

Each training example was converted into a structured format to provide the model with broader context around the buggy line. Four key fields were included:

  • [CONTEXT] ‚Äì Nearby lines of code (before/after)
  • [SNIPPET] ‚Äì The line suspected to be buggy
  • [COMMIT] ‚Äì Commit message associated with the fix
  • [PARENT] ‚Äì Commit message from the previous commit

Identifiers were masked (e.g. VAR1, VAR2) to reduce bias and improve model generalisation across different projects.

Figure: Example of identifier masking used in the input formatting step. Variable names are replaced with placeholders like VAR1 to avoid overfitting on specific projects.

Model & Training Pipeline

The model is based on CodeBERT, a pre-trained transformer with 12 layers. For this project, the full transformer was frozen except for the embedding layer, allowing it to adapt to custom special tokens like [CONTEXT] and [SNIPPET].

A custom classification head was added on top, consisting of two linear layers with ReLU activation and Dropout for regularisation. The model outputs a binary prediction indicating whether the input line is buggy or not.

Figure: Internal architecture of the model. Enhanced inputs are tokenised, passed through a partially frozen CodeBERT encoder, then through a custom classification head to generate a bug prediction.

To prevent overfitting, a custom MultiEvalTrainer class was used to monitor both validation loss and performance on a held-out portion of the training data. Training was capped at 50 epochs with early stopping triggered by stagnating validation F1.

Evaluation Metrics

The final model was evaluated on the test split from the repetition subset. It achieved strong overall performance, especially in recall ‚Äî successfully catching the majority of buggy lines.

  • F1 Score: 0.80
  • Precision: 0.73
  • Recall: 0.87

CLI Prototype Integration

A CLI prototype was developed to simulate how this model could fit into real software development workflows. The tool intercepts staged diffs, builds structured model input, and returns a real-time prediction before the code is committed.

It uses a Git hook + Flask API setup and is fully containerised with Docker, allowing for easy deployment and testing.

Evaluation Summary

ManySStuBs4J ‚Äî Unique Split

The model generalised well to the held-out unique split, achieving 74% Top-1 Accuracy and an F1 score of 0.77. These are promising results given the unseen nature of these bugs and the fact that the model was only trained on repeated patterns.

Defects4J Evaluation

To assess real-world generalisability, the model was evaluated on 30 single-line bug examples manually extracted from Defects4J. This required converting each bug fix into the exact format expected by the model ‚Äî including [CONTEXT], [SNIPPET], [COMMIT], and [PARENT] fields, along with identifier masking.

Many existing studies that report Defects4J results don‚Äôt specify how they filtered or preprocessed bugs, leaving ambiguity about whether only trivial cases were selected or how bug boundaries were defined. This makes direct comparison slightly unreliable ‚Äî but still informative.

Even with these limitations, the model achieved 62% Top-1 Accuracy and an F1 score of 0.76 ‚Äî a strong result, especially considering no training was done on this dataset.

Top-1 Accuracy refers to whether the model‚Äôs highest-confidence prediction is correct ‚Äî and is a standard metric in bug localisation. Since this model outputs one prediction per input line, Top-1 Accuracy is entirely valid and comparable with existing studies.

MRR and MAP were not computed, as these metrics assume ranked candidate lists or search over an entire file. In contrast, this model performs binary classification per line ‚Äî a different formulation. A ranking-based extension could be added in future work.

Figure: Comparison of top-1 accuracy (%) across various bug localisation models and datasets.

Many of the studies compared here rely on multi-line, file-level, or ranked prediction setups, often using reinforcement learning, complex search spaces, or repair objectives. Despite this, the classification-only model in this project:

  • Outperformed Mashhadi et al. on the exact same unique subset of ManySStuBs4J (74% vs. 23.7%)
  • Achieved comparable results on Defects4J despite no fine-tuning or dataset-specific tuning

Overall, the results support the research question: a context-aware, classification-based CodeBERT model trained solely on static, pre-commit information can effectively localise single-line Java bugs and even outperform more complex repair or ranking-based systems on benchmark datasets. However, these findings should be interpreted as a proof of concept rather than a fully generalisable solution. The model‚Äôs success in a constrained, single-line classification setting is promising, but broader applicability ‚Äî especially at the file or multi-line level ‚Äî would require further training, architectural adaptations, and richer datasets. This project demonstrates clear early-stage feasibility and lays a foundation for future work in lightweight, pre-commit bug localisation.

‚ö†Ô∏è Threats to Validity
  • Granularity: This project focuses on single-line bug detection. Results from multi-line or hunk-level studies may not be directly comparable.
  • Dataset Filtering: The exact bug filtering process used in other studies is often unclear, making it hard to ensure comparisons are apples-to-apples.
  • Binary Framing: A yes/no decision per line avoids ranking, which limits the use of metrics like MAP/MRR.
  • Tool Adoption: While a working CLI prototype was built, usability and trust still need to be tested with real developers in longer-term settings.

In summary, this evaluation shows that even a lightweight model, when carefully fine-tuned and fed rich static context, can generalise across datasets and outperform more complex localisation tools ‚Äî making it both practical and effective for real-world pre-commit scenarios.
